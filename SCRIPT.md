# 🎬 BEHAVIOR & INTERACTION SCRIPT: WAR RADAR MODULE

## 👥 1. USER OPERATIONAL FLOW (Сценарий работы пользователя)

1. **Инициализация:** Пользователь вводит свой Windows PC Username в инпут для генерации точного локального пути к логам игры. При необходимости нажимает `[Copy Path]`.
2. **Первичный анализ:** Пользователь выбирает актуальный файл рыночного лога из папки игры (`.txt` / `.csv`) и нажимает кнопку **«Analyze Market Log»**.
3. **Реакция на алерты:**
    - Интерфейс выводит таблицы ордеров. Строки со статусом `OUTBID` сигнализируют о том, что ставка игрока перебита.
    - Пользователь нажимает кнопку `[Copy]` напротив цены. Скрипт копирует модифицированную цену во внутренний буфер обмена.
    - **Правило модификации цены:** Направление изменения цены и точный конкурентный шаг вычисляются строго через утилиту `calculateModifiedIsk()` (сдвиг для Sell-ордеров идет в `'minus'`, для Buy-ордеров — в `'plus'`).
    - Пользователь переставляет цену в клиенте игры EVE Online, просто нажав `Ctrl+V` в игровом поле ввода.
4. **Маркировка выполнения:**
    - Сразу после изменения цены в игре, пользователь **вручную ставит нативную галочку** в колонке `changed` напротив обработанного ордера.
    - **Ожидаемое поведение UI:** Строка мгновенно блекнет (`opacity: 0.35`), визуально разгружая внимание трейдера. При этом галочка физически остается внутри чекбокса. Пользователь имеет право снять галочку назад, если ошибся — строка вернет яркость `1.0`.
5. **Активация радара:** Пользователь нажимает **«Start Auto-Radar»**. Модуль переходит в автономный фоновый режим.

## 🛰️ 2. FUNCTIONAL RADAR ENGINE SCENARIOS (Скрипты функций и фоновых процессов)

### Сценарий А: Фоновое сканирование (Web Worker Anti-Throttling)

- При включении Авто-Радара запускается изолированный Web Worker.
- Каждые **5 минут 05 секунд** (окно сброса кэша ESI CCP Games) Worker инициирует сетевой запрос к серверному эндпоинту `/api/war`.
- **Правило передачи данных:** Фоновый чекер **всегда отправляет на сервер весь исходный текст лога целиком**. Он игнорирует тот факт, стоят ли сейчас у пользователя какие-то галочки в интерфейсе или нет (сервер должен знать полную картину стакана).

### Сценарий Б: Сброс состояний при обновлении данных (Data Hydration Reset)

- Когда сервер возвращает свежие результаты сканирования стаканов, клиентский поток React обновляет состояние `warItems`.
- В этот момент срабатывает **скрипт автосброса**:
    1. Все нативные чекбоксы в таблице полностью размонтируются и создаются заново. Все галочки **автоматически сбрасываются в состояние false**.
    2. Все строки таблиц (`<tr>`), которые были затушеваны пользователем, принудительно возвращают свою нативную визуальную яркость (`opacity: 1`).
    3. Война начинается заново на основе свежих рыночных срезов.

## 🚨 3. OS-LEVEL NOTIFICATION SCENARIO (Моргание вкладки браузера)

Функция контроля заголовка страницы (`document.title`) работает в реальном времени, непрерывно анализируя физическое состояние элементов в DOM:

1. **Trigger включения:** Если в таблицах есть хотя бы один ордер со статусом `OUTBID`, у которого **НЕ стоит галочка выполнения** и статус **НЕ изменен вручную на `IGNORED`**, запускается таймер. Вкладка начинает моргать раз в 1 секунду, чередуя заголовки `🔴 0.01 ISK WAR` и `ISK Master`. Это привлекает внимание пользователя, даже если он смотрит фильм или играет в игру.
2. **Условия затухания алертов:** Моргание вкладки должно **моментально прекратиться** (заголовок жестко фиксируется как `ISK Master`), если выполнено хотя бы одно из условий для _всех_ проблемных строк:
    - Пользователь проставил на строку галочку выполнения (строка поблекла).
    - Пользователь вручную кликнул на кнопку статуса и перевел её из `OUTBID` в состояние `IGNORED`.
    - Авто-радар сделал новый виток сканирования и сервер сообщил, что конкуренты ушли (статус стал пустым `''`).
