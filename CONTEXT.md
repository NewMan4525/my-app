# === КОНТЕКСТ ПРОЕКТА: ISK MASTER (NEXT.JS APP ROUTER + TYPESCRIPT) ===

## 1. СУТЬ И АРХИТЕКТУРА ПРОЕКТА

- Фронтенд торгового симулятора / калькулятора вселенной EVE Online на базе React 19.2.4 и Next.js 16.2.4.
- Строгий принцип Multi-Page Application (MPA). Смена страниц полностью сбрасывает стейт React в памяти.
- Логика работы: Пользователь настраивает параметры в формах, нажимает кнопку "OK" (сохранение в LocalStorage). По кнопке "Get Data" клиент собирает объекты настроек (trade_settings, user_stats, user_skills) и отправляет их на сервер POST-запросом в формате JSON (/api/buy).
- Динамики и реактивности "на лету" после получения данных на клиенте нет (кроме темы, сортировки и выделения строк таблицы). Сервер возвращает полностью готовый массив просчитанных данных для таблицы ордеров (тип IMarketItem).
- Жесткие правила линтинга (strict: true): ПОЛНОСТЬЮ ЗАПРЕЩЕН тип "any" (строгая типизация интерфейсов), ЗАПРЕЩЕН синхронный "setState" внутри "useEffect" во избежание каскадных рендеров и ошибок гидратации.
- Исключены нативные блокирующие уведомления "alert". Все ошибки логируются через console.error.

## 3. ТЕКУЩИЙ СТАТУС И РЕАЛИЗОВАННЫЙ ФУНКЦИОНАЛ

- **Изоляция бэкенда и общий кэш:** Реализован Вариант Б. База данных SQLite (esi_cache.db) хранит исключительно 100% сырые, полные и нефильтрованные ответы от ESI API (ордера региона на 5 минут, история и типы на 24 часа по токенам Expires/ETag). Пользовательская фильтрация изолирована для каждого запроса и выполняется в оперативной памяти в самом конце конвейера. Разные фильтры пользователей не загрязняют общую базу.
- **Уничтожение сетевых затыков:** HEAD-запросы полностью удалены из execute.ts. Первая страница ордеров региона скачивается обычным GET через queryHandler с p-limit(20). Из заголовков извлекается количество страниц x-pages, а контент сразу пускается в обработку без дублирования запроса. Сбор истории торгов идет точечно поштучно строго для ID предметов, реально выставленных в хабе и радиусе 1 прыжка. Повторные запросы к прогретой базе летят за 1-2 секунды вместо 103 секунд холодного старта.
- **Математика торговых маршрутов EVE Online:** В profitCalculations.ts расчет разделен на два независимых шага на основе параметров marketPlaceBuyIsCitadel (закупка) и marketPlaceSellIsCitadel (продажа). Симулируется закупка в Цитадели (0.5% налог SCC + комиссия владельца структуры + 100 ISK) и вывоз на продажу на NPC-хаб (Sales Tax через навык accounting и брокерский сбор через навыки broker_relationship и стендинги factionStand/stationOwnerStand конкретного региона из констант HUBS). Внедрено жесткое округление сотых долей ISK Math.round(val \* 100) / 100 на каждом этапе для устранения погрешности чисел с плавающей точкой в JS. Стендинги в конвейере извлекаются из HUBS по фракционным алиасам, исправляя баг нулевых отношений.
- **Синхронизация стейта форм и радио-кнопок:** Восстановлена обратная совместимость классов в classes.ts (суффиксы индексов сохранены для number-полей). В inputsBlock.tsx для типа radio значение теперь ищется по реальному inputName конкретной обрабатываемой группы, а не первой в массиве. В userForm.tsx ключи стейта полностью засинхронизированы с индексами (skills_accounting2, skills_broker_relationship0), что намертво починило сохранение и отображение скиллов в LocalStorage. В OptionsBar радио-группы Buy Station (marketplace1) и Sell Station (marketplace2) работают независимо.
- **Защита от Hydration Error без useEffect:** Использование динамического импорта { ssr: false } для UserForm и OptionsBar позволило useState лениво парсить localStorage напрямую в момент инициализации в браузере. В коде нет эффектов и вызовов setState внутри них, блокирующих компиляцию ESLint (react-hooks/set-state-in-effect). На странице Buy стейт инициализируется пустой строкой [], а подгрузка sessionStorage выполнена асинхронно через setTimeout(..., 0) в useEffect после полной гидратации, что полностью убрало падение React 19 в Hydration Mismatch.
- **Сессионное кэширование таблицы ордеров:** Тяжелый массив готовых данных от сервера (IMarketItem[]) при POST-запросе пишется в sessionStorage под ключом cached_market_items. При переходах пользователя между страницами /buy и /user в рамках одной вкладки таблица восстанавливается мгновенно за 0 мс, не обнуляясь и не нагружая бэкенд повторными расчетами. При закрытии вкладки память диска очищается автоматически. Объем 5000 строк (~750 КБ) не превышает лимиты сессии.
- **Универсальная таблица Table<T>:** Реализована Windows-механика множественного выделения строк. Клик с Ctrl точечно инвертирует стейт. Клик с Shift полностью перезаписывает диапазон ID от последнего выбранного до текущего по массиву sortedItems, позволяя как расширять, так и сужать выделение. При вызове handleSort якорный ID сбрасывается для защиты от хаотичной подсветки. Добавлено e.preventDefault() для Shift и инлайн userSelect: 'none' для строк, что полностью заблокировало синее выделение текста ячеек браузером. Кнопки [Copy] изолированы через e.stopPropagation() и меняют текст на [Copied] через DOM.
